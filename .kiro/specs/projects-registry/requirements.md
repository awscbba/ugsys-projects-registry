# Requirements Document — ugsys-projects-registry

## Introduction

`ugsys-projects-registry` is Phase 2 of the ugsys platform. It extracts the project catalog and volunteer subscription management from the Registry monolith into a standalone Python microservice following the same hexagonal architecture as `ugsys-identity-manager` and `ugsys-user-profile-service`.

The service owns three bounded contexts: **project catalog** (CRUD, publish/unpublish, status lifecycle), **volunteer subscriptions** (apply, approve, reject, cancel with dynamic form data), and **dynamic forms** (per-project custom fields). It publishes domain events to EventBridge and consumes identity events to keep its user-related state consistent.

The service is serverless (Lambda + API Gateway via Mangum), uses DynamoDB for persistence, and integrates with `ugsys-identity-manager` via S2S tokens for user creation in public flows.

---

## Glossary

- **Project_Registry**: The `ugsys-projects-registry` service as a whole.
- **Project**: A community initiative with a defined lifecycle (`pending` → `active` → `completed` | `cancelled`).
- **Subscription**: A volunteer's enrollment in a Project, with its own approval lifecycle (`pending` → `active` | `rejected`; `active` → `cancelled`).
- **FormSchema**: A per-project collection of up to 20 custom fields used to collect structured data from subscribers.
- **FormSubmission**: A subscriber's responses to a Project's FormSchema.
- **ProjectImage**: An image asset associated with a Project, stored in S3 and served via CloudFront.
- **Subscriber**: An authenticated user who has created a Subscription for a Project.
- **Admin**: A user with role `admin`, `super_admin`, or `moderator` in the JWT `roles` claim.
- **Super_Admin**: A user with role `super_admin` — subscriptions are auto-approved for this role.
- **Identity_Manager**: The `ugsys-identity-manager` service, called via S2S token for user creation.
- **Event_Publisher**: The `ugsys-event-lib` `EventPublisher` component that publishes events to `ugsys-platform-bus`.
- **Event_Consumer**: The Lambda handler that processes EventBridge events from `ugsys-platform-bus`.
- **ULID**: Universally Unique Lexicographically Sortable Identifier — used for all IDs in this service.
- **S2S_Token**: A service-to-service JWT obtained via `client_credentials` grant from Identity_Manager.

## Requirements

### Requirement 1: Service Scaffold and Cross-Cutting Infrastructure

**User Story:** As a platform engineer, I want the service to follow the canonical hexagonal architecture and security standards, so that it integrates consistently with the rest of the ugsys platform.

#### Acceptance Criteria

1. THE Project_Registry SHALL expose all API routes under the `/api/v1/` prefix, with the exception of `GET /health` and `GET /`.
2. THE Project_Registry SHALL apply middleware in this order: `CorrelationIdMiddleware`, `SecurityHeadersMiddleware`, `RateLimitMiddleware` (60 req/min per authenticated user, fallback to IP for unauthenticated).
3. THE Project_Registry SHALL use `ugsys-auth-client` for JWT RS256 validation on all protected endpoints, rejecting HS256 and `none` algorithm tokens with HTTP 401.
4. THE Project_Registry SHALL wrap all successful API responses in the envelope `{ "data": ..., "meta": { "request_id": "..." } }`.
5. THE Project_Registry SHALL wrap all error responses in the envelope `{ "error": "<error_code>", "message": "<user_safe_message>", "data": {} }`, where `message` never contains internal details, stack traces, or PII.
6. THE Project_Registry SHALL use ULIDs (not UUIDs) for all entity IDs generated by this service.
7. THE Project_Registry SHALL use `ugsys-logging-lib` `configure_logging()` as the first call in `src/main.py`, and use `structlog` exclusively — no `print()` or `logging.getLogger()`.
8. THE Project_Registry SHALL log `duration_ms` on every application service method boundary.
9. THE Project_Registry SHALL raise only domain exceptions from `src/domain/exceptions.py` in application and domain layers — never raw `Exception`, `ValueError`, or `HTTPException`.
10. THE Project_Registry SHALL serve as a Mangum-wrapped Lambda handler, with `handler.handler` as the entry point.
11. WHEN the service starts, THE Project_Registry SHALL register itself with Identity_Manager via S2S token (non-fatal — log warning and continue if unreachable).
12. WHEN the service starts, THE Project_Registry SHALL fetch operator config from Identity_Manager and apply it over env var defaults (non-fatal — use env var defaults if unreachable).

### Requirement 2: Project Catalog — CRUD

**User Story:** As a community organizer, I want to create and manage projects, so that volunteers can discover and subscribe to community initiatives.

#### Acceptance Criteria

1. WHEN a request to `POST /api/v1/projects` is received with a valid JWT and a valid project payload, THE Project_Registry SHALL create a Project with a new ULID, set `status` to `pending`, set `current_participants` to `0`, and return HTTP 201 with the full project object.
2. WHEN a project is created with `end_date` earlier than `start_date`, THE Project_Registry SHALL return HTTP 422 with error code `INVALID_DATE_RANGE`.
3. WHEN a project is created with `max_participants` less than 1, THE Project_Registry SHALL return HTTP 422 with error code `INVALID_MAX_PARTICIPANTS`.
4. WHEN a request to `GET /api/v1/projects` is received with a valid Admin JWT, THE Project_Registry SHALL return a paginated list of all projects, supporting query parameters `limit`, `status`, and `category`.
5. WHEN a request to `GET /api/v1/projects/public` is received without authentication, THE Project_Registry SHALL return only projects where `status = active` AND `is_enabled = true`, excluding `notification_emails` and other internal fields.
6. WHEN a request to `GET /api/v1/projects/{id}` is received with a valid JWT, THE Project_Registry SHALL return the full project object if it exists, or HTTP 404 with error code `PROJECT_NOT_FOUND`.
7. WHEN a request to `PUT /api/v1/projects/{id}` is received from the project owner or an Admin, THE Project_Registry SHALL update the specified fields and return the updated project.
8. WHEN a request to `PUT /api/v1/projects/{id}` is received from a user who is neither the project owner nor an Admin, THE Project_Registry SHALL return HTTP 403 with error code `FORBIDDEN`.
9. WHEN a request to `DELETE /api/v1/projects/{id}` is received with a valid Admin JWT, THE Project_Registry SHALL delete the project and return HTTP 204.
10. WHEN a project's `status` is updated to `completed` or `cancelled`, THE Project_Registry SHALL update all `active` Subscriptions for that project to match the new project status.
11. THE Project_Registry SHALL store projects in DynamoDB table `ugsys-projects-{env}` with partition key `PROJECT#{id}`, sort key `PROJECT`, a GSI on `status` (PK) + `created_at` (SK), and a GSI on `created_by`.

### Requirement 3: Project Publish/Unpublish Workflow

**User Story:** As an admin, I want to control project visibility, so that only ready projects are shown to the public.

#### Acceptance Criteria

1. WHEN a project's `status` is updated to `active` via `PUT /api/v1/projects/{id}`, THE Project_Registry SHALL set `is_enabled` to `true` if not explicitly provided, and publish a `projects.project.published` event.
2. WHEN a project's `status` is updated to `cancelled` or `completed`, THE Project_Registry SHALL set `is_enabled` to `false`.
3. WHILE a project has `status = pending`, THE Project_Registry SHALL exclude it from `GET /api/v1/projects/public` responses.
4. WHEN a project's `is_enabled` field is set to `false` via `PUT /api/v1/projects/{id}`, THE Project_Registry SHALL exclude it from `GET /api/v1/projects/public` responses regardless of `status`.

### Requirement 4: Volunteer Subscriptions — Apply, Approve, Reject, Cancel

**User Story:** As a volunteer, I want to subscribe to a project and track my application status, so that I can participate in community initiatives.

#### Acceptance Criteria

1. WHEN a request to `POST /api/v1/projects/{id}/subscriptions` is received with a valid JWT, THE Project_Registry SHALL create a Subscription with a new ULID and set `status` to `pending`, unless the requester has role `super_admin`, in which case `status` SHALL be set to `active`.
2. WHEN a Subscription is created with `status = active` (super_admin auto-approval), THE Project_Registry SHALL increment `current_participants` on the associated Project by 1.
3. WHEN a duplicate Subscription is detected (same `person_id` + `project_id` already exists), THE Project_Registry SHALL return HTTP 409 with error code `SUBSCRIPTION_ALREADY_EXISTS`.
4. WHEN a request to `PUT /api/v1/projects/{id}/subscribers/{sub_id}` is received with a valid Admin JWT and `status = active`, THE Project_Registry SHALL approve the Subscription, increment `current_participants` on the Project by 1, and publish a `projects.subscription.approved` event.
5. WHEN a request to `PUT /api/v1/projects/{id}/subscribers/{sub_id}` is received with a valid Admin JWT and `status = rejected`, THE Project_Registry SHALL reject the Subscription and publish a `projects.subscription.rejected` event.
6. WHEN a request to `DELETE /api/v1/projects/{id}/subscribers/{sub_id}` is received from the Subscription owner or an Admin, THE Project_Registry SHALL cancel the Subscription, decrement `current_participants` on the Project if the Subscription was `active`, and publish a `projects.subscription.cancelled` event.
7. WHEN a request to `DELETE /api/v1/projects/{id}/subscribers/{sub_id}` is received from a user who is neither the Subscription owner nor an Admin, THE Project_Registry SHALL return HTTP 403 with error code `FORBIDDEN`.
8. WHEN `enable_subscription_notifications` is `true` on a Project and a new Subscription is created, THE Project_Registry SHALL publish a `projects.subscription.created` event with the project's `notification_emails` in the payload so that the omnichannel service can send notifications.
9. THE Project_Registry SHALL store subscriptions in DynamoDB table `ugsys-subscriptions-{env}` with partition key `SUB#{id}`, sort key `SUB`, a GSI on `person_id` (PK) + `created_at` (SK), a GSI on `project_id` (PK) + `created_at` (SK), and a GSI on `person_id#project_id` for duplicate detection.
10. WHEN a request to `GET /api/v1/projects/{id}/subscriptions` is received with a valid Admin or Moderator JWT, THE Project_Registry SHALL return a paginated list of all Subscriptions for that project.
11. WHEN a request to `GET /api/v1/subscriptions/person/{person_id}` is received from the person themselves or an Admin, THE Project_Registry SHALL return enriched Subscriptions including `project_name`, `project_description`, `project_status`, `person_name`, and `person_email`.

### Requirement 5: Dynamic Forms — Schema Management

**User Story:** As a project organizer, I want to attach a custom form to my project, so that I can collect structured information from subscribers.

#### Acceptance Criteria

1. WHEN a request to `PUT /api/v1/projects/{id}/form-schema` is received from the project owner or an Admin with a valid FormSchema payload, THE Project_Registry SHALL validate and store the schema on the Project.
2. WHEN a FormSchema contains more than 20 fields, THE Project_Registry SHALL return HTTP 422 with error code `FORM_SCHEMA_TOO_MANY_FIELDS`.
3. WHEN a FormSchema contains duplicate field IDs, THE Project_Registry SHALL return HTTP 422 with error code `FORM_SCHEMA_DUPLICATE_FIELD_IDS`.
4. WHEN a FormSchema field of type `poll_single` or `poll_multiple` has fewer than 2 or more than 10 options, THE Project_Registry SHALL return HTTP 422 with error code `FORM_SCHEMA_INVALID_OPTIONS`.
5. WHEN the serialized FormSchema JSON exceeds 50KB, THE Project_Registry SHALL return HTTP 422 with error code `FORM_SCHEMA_TOO_LARGE`.
6. WHEN a request to `GET /api/v1/projects/{id}/enhanced` is received with a valid JWT, THE Project_Registry SHALL return the full project object including the `form_schema` field.
7. WHEN a request to `POST /api/v1/projects/enhanced` is received with a valid JWT, THE Project_Registry SHALL create a Project and attach the provided FormSchema in a single operation.

### Requirement 6: Dynamic Forms — Submission and Validation

**User Story:** As a volunteer, I want to submit my responses to a project's custom form, so that the organizer has the information they need to evaluate my application.

#### Acceptance Criteria

1. WHEN a request to `POST /api/v1/form-submissions` is received with a valid JWT and a valid submission payload, THE Project_Registry SHALL validate the responses against the Project's FormSchema and store the FormSubmission with a new ULID.
2. WHEN a FormSubmission response for a `poll_single` field is not a string matching one of the field's `options`, THE Project_Registry SHALL return HTTP 422 with error code `FORM_SUBMISSION_INVALID_RESPONSE`.
3. WHEN a FormSubmission response for a `poll_multiple` field is not a list where every item is in the field's `options`, THE Project_Registry SHALL return HTTP 422 with error code `FORM_SUBMISSION_INVALID_RESPONSE`.
4. WHEN a FormSubmission omits a field where `required = true`, THE Project_Registry SHALL return HTTP 422 with error code `FORM_SUBMISSION_MISSING_REQUIRED_FIELD`.
5. WHEN a FormSubmission is submitted for a Project that has no FormSchema, THE Project_Registry SHALL return HTTP 422 with error code `PROJECT_HAS_NO_FORM_SCHEMA`.
6. WHEN a request to `GET /api/v1/form-submissions/project/{project_id}` is received with a valid Admin JWT, THE Project_Registry SHALL return all FormSubmissions for that project.
7. WHEN a request to `GET /api/v1/form-submissions/person/{person_id}/project/{project_id}` is received from the person themselves or an Admin, THE Project_Registry SHALL return the FormSubmission for that person and project, or HTTP 404 if none exists.
8. THE Project_Registry SHALL store FormSubmissions in DynamoDB table `ugsys-form-submissions-{env}` with partition key `SUBMISSION#{id}`, sort key `SUBMISSION`, a GSI on `project_id`, and a GSI on `person_id`.

#### Parser/Serializer Round-Trip Property

9. FOR ALL valid FormSchema objects, THE Project_Registry SHALL serialize a FormSchema to JSON and deserialize it back to an equivalent FormSchema object (round-trip property: `deserialize(serialize(schema)) == schema`).

### Requirement 7: Public Endpoints — No-Auth Registration and Subscription

**User Story:** As a community member without an account, I want to register and subscribe to a project in a single step, so that I can participate without navigating a separate sign-up flow.

#### Acceptance Criteria

1. WHEN a request to `POST /api/v1/public/check-email` is received with an email address, THE Project_Registry SHALL return `{ "exists": true }` if the email is registered in Identity_Manager, or `{ "exists": false }` otherwise — without requiring authentication.
2. WHEN a request to `POST /api/v1/public/register` is received with valid personal data and the email does not exist, THE Project_Registry SHALL call Identity_Manager via S2S_Token to create the user account and return HTTP 201 with `{ "personId": "...", "email": "...", "emailSent": true, "message": "..." }`.
3. WHEN a request to `POST /api/v1/public/register` is received and the email already exists in Identity_Manager, THE Project_Registry SHALL return HTTP 400 with error code `EMAIL_ALREADY_EXISTS`.
4. WHEN a request to `POST /api/v1/public/subscribe` is received and the person does not exist, THE Project_Registry SHALL call Identity_Manager via S2S_Token to create the user account, then create a Subscription with `status = pending`, and return HTTP 201.
5. WHEN a request to `POST /api/v1/public/subscribe` is received and the person already exists but has no Subscription for the given project, THE Project_Registry SHALL create a Subscription with `status = pending` and return HTTP 201.
6. WHEN a request to `POST /api/v1/public/subscribe` is received and the person already has a Subscription for the given project, THE Project_Registry SHALL return HTTP 409 with error code `SUBSCRIPTION_ALREADY_EXISTS`.
7. THE Project_Registry SHALL set all Subscriptions created via `POST /api/v1/public/subscribe` to `status = pending` regardless of the requester's role, since no JWT is present.
8. WHEN a Subscription is created via `POST /api/v1/public/subscribe`, THE Project_Registry SHALL publish a `projects.subscription.created` event with `notification_emails` from the Project in the payload.
9. WHEN a request to `POST /api/v1/subscriptions/check` is received with `personId` and `projectId`, THE Project_Registry SHALL return `{ "data": { "exists": true } }` if a Subscription exists for that pair, or `{ "data": { "exists": false } }` otherwise.

### Requirement 8: Image Upload via Presigned S3 URLs

**User Story:** As a project organizer, I want to upload images for my project, so that the project listing is visually appealing.

#### Acceptance Criteria

1. WHEN a request to `POST /api/v1/images/upload-url` is received with a valid JWT, a filename, a content type, and a file size, THE Project_Registry SHALL generate and return a presigned S3 upload URL, an `image_id` (ULID), and a CloudFront URL in the format `https://cdn.cbba.cloud.org.bo/{filename}`.
2. WHEN the requested `file_size` exceeds 10MB, THE Project_Registry SHALL return HTTP 422 with error code `IMAGE_TOO_LARGE`.
3. WHEN the requested `content_type` is not one of `image/jpeg`, `image/png`, `image/gif`, or `image/webp`, THE Project_Registry SHALL return HTTP 422 with error code `IMAGE_INVALID_CONTENT_TYPE`.
4. THE Project_Registry SHALL generate presigned S3 URLs targeting the bucket `ugsys-images-{env}` with a TTL of 300 seconds.

### Requirement 9: Admin Panel Endpoints

**User Story:** As an admin, I want a dedicated admin surface with dashboard data, bulk user actions, and paginated views, so that I can manage the platform efficiently.

#### Acceptance Criteria

1. WHEN a request to `GET /api/v1/admin/dashboard` is received with a valid Admin JWT, THE Project_Registry SHALL return `{ "totalUsers": N, "activeUsers": N, "totalProjects": N, "activeProjects": N, "totalSubscriptions": N, "activeSubscriptions": N, "lastUpdated": "..." }`.
2. WHEN a request to `GET /api/v1/admin/dashboard/enhanced` is received with a valid Admin JWT, THE Project_Registry SHALL return the basic dashboard data plus `adminUsers`, `recentSignups`, `projectStats` (per-project subscription counts), and `systemHealth`.
3. WHEN a request to `GET /api/v1/admin/analytics` is received with a valid Admin JWT, THE Project_Registry SHALL return aggregated counts for users, projects, and subscriptions broken down by status.
4. WHEN a request to `GET /api/v1/admin/users/paginated` is received with a valid Admin JWT, THE Project_Registry SHALL return a paginated user list supporting query parameters `page`, `pageSize`, `sortBy`, `sortDirection`, `search`, `isAdmin`, `isActive`, and `emailVerified`.
5. WHEN a request to `POST /api/v1/admin/users/bulk-action` is received with a valid Admin JWT, a valid `action` (`activate`, `deactivate`, or `delete`), and a non-empty `userIds` list, THE Project_Registry SHALL execute the action for each user and return `{ "action": "...", "totalUsers": N, "successCount": N, "failureCount": N, "results": [ { "userId": "...", "status": "success | failed", "error": "..." } ] }`.
6. WHEN a bulk `delete` action is requested for a user who has `active` or `pending` Subscriptions, THE Project_Registry SHALL return `status = failed` for that user with error code `BUSINESS_RULE_VIOLATION` and continue processing the remaining users.
7. WHEN a request to `GET /api/v1/admin/subscriptions` is received with a valid Admin JWT, THE Project_Registry SHALL return a paginated list of all Subscriptions across all projects.
8. WHEN a request to `POST /api/v1/admin/users/bulk-action` is received with a missing `action` or empty `userIds`, THE Project_Registry SHALL return HTTP 400 with error code `VALIDATION_ERROR`.

### Requirement 10: EventBridge — Publishing Domain Events

**User Story:** As a platform engineer, I want the service to publish domain events to EventBridge, so that downstream services (omnichannel, mass-messaging) can react to project and subscription changes.

#### Acceptance Criteria

1. WHEN a Project is created, THE Event_Publisher SHALL publish a `projects.project.created` event to `ugsys-platform-bus` with source `ugsys.projects-registry`.
2. WHEN a Project is updated, THE Event_Publisher SHALL publish a `projects.project.updated` event to `ugsys-platform-bus`.
3. WHEN a Project's `status` transitions to `active`, THE Event_Publisher SHALL publish a `projects.project.published` event to `ugsys-platform-bus`.
4. WHEN a Project is deleted, THE Event_Publisher SHALL publish a `projects.project.deleted` event to `ugsys-platform-bus`.
5. WHEN a Subscription is created, THE Event_Publisher SHALL publish a `projects.subscription.created` event to `ugsys-platform-bus`, including `notification_emails` from the Project in the payload.
6. WHEN a Subscription is approved (status → `active`), THE Event_Publisher SHALL publish a `projects.subscription.approved` event to `ugsys-platform-bus`.
7. WHEN a Subscription is rejected, THE Event_Publisher SHALL publish a `projects.subscription.rejected` event to `ugsys-platform-bus`.
8. WHEN a Subscription is cancelled, THE Event_Publisher SHALL publish a `projects.subscription.cancelled` event to `ugsys-platform-bus`.
9. THE Event_Publisher SHALL use `ugsys-event-lib` `EventPublisher` for all event publishing — no direct `boto3` calls from application or domain layers.
10. IF an event publish call fails, THE Project_Registry SHALL log the failure with `structlog` at `error` level including the event type and correlation ID, and SHALL NOT roll back the primary database operation.

### Requirement 11: EventBridge — Consuming Identity Events

**User Story:** As a platform engineer, I want the service to react to identity events, so that user deactivation is reflected consistently in subscription state.

#### Acceptance Criteria

1. WHEN the Event_Consumer receives an `identity.user.deactivated` event, THE Project_Registry SHALL cancel all `active` and `pending` Subscriptions for the deactivated user and publish a `projects.subscription.cancelled` event for each cancelled Subscription.
2. WHEN the Event_Consumer receives an `identity.user.deactivated` event for a user with no Subscriptions, THE Project_Registry SHALL process the event successfully and take no further action.
3. IF the Event_Consumer fails to process an `identity.user.deactivated` event due to a transient error, THE Project_Registry SHALL allow the event to be retried via EventBridge's retry policy — it SHALL NOT silently swallow the error.
4. THE Event_Consumer SHALL log every received event at `info` level with the event type, source, and correlation ID before processing.

### Requirement 12: Authorization and IDOR Prevention

**User Story:** As a security engineer, I want all resource access to be scoped to the authenticated user's identity and role, so that users cannot access or modify resources they don't own.

#### Acceptance Criteria

1. THE Project_Registry SHALL extract the requester's identity exclusively from the validated JWT `sub` claim — never from a client-provided user ID in the request body or path.
2. WHEN a non-Admin user requests a resource owned by a different user, THE Project_Registry SHALL return HTTP 403 with error code `FORBIDDEN` — never HTTP 404 (which would leak existence information for most resources).
3. WHEN a JWT is absent or invalid on a protected endpoint, THE Project_Registry SHALL return HTTP 401 with a generic `"Invalid or expired token"` user message — never exposing JWT validation details.
4. THE Project_Registry SHALL validate the `aud` claim against the service's configured Cognito client ID and the `iss` claim against the configured Cognito User Pool URL on every token validation.
5. WHEN a JWT contains an algorithm other than RS256, THE Project_Registry SHALL reject it with HTTP 401 before attempting signature verification.
6. THE Project_Registry SHALL enforce that `GET /api/v1/subscriptions/person/{person_id}` returns HTTP 403 when the requester's JWT `sub` does not match `person_id` and the requester is not an Admin.

### Requirement 13: Input Validation and Security

**User Story:** As a security engineer, I want all inputs to be validated and sanitized before reaching business logic, so that the service is protected against injection and oversized payload attacks.

#### Acceptance Criteria

1. THE Project_Registry SHALL validate all request bodies via Pydantic v2 models before the request reaches the application layer.
2. THE Project_Registry SHALL enforce a maximum request body size of 1MB via middleware, returning HTTP 413 for oversized requests.
3. WHEN a `Project` name exceeds 200 characters, THE Project_Registry SHALL return HTTP 422 with error code `VALIDATION_ERROR`.
4. WHEN a `Project` description exceeds 5000 characters, THE Project_Registry SHALL return HTTP 422 with error code `VALIDATION_ERROR`.
5. WHEN a `Project` `rich_text` field exceeds 10000 characters (approximately 15KB), THE Project_Registry SHALL return HTTP 422 with error code `VALIDATION_ERROR`.
6. WHEN a `Subscription` `notes` field exceeds 1000 characters, THE Project_Registry SHALL return HTTP 422 with error code `VALIDATION_ERROR`.
7. WHEN a `FormSchema` field `question` exceeds 500 characters, THE Project_Registry SHALL return HTTP 422 with error code `VALIDATION_ERROR`.
8. THE Project_Registry SHALL sanitize all free-text string inputs using `html.escape()` before persisting to DynamoDB.
9. THE Project_Registry SHALL never log request body contents, JWT tokens, or user-provided free-text fields — only structured identifiers such as `project_id`, `person_id`, and `subscription_id`.

### Requirement 14: Data Migration from Registry PostgreSQL

**User Story:** As a platform engineer, I want existing Registry data migrated to DynamoDB, so that no community data is lost during the transition from the monolith.

#### Acceptance Criteria

1. THE Project_Registry SHALL provide a migration script at `scripts/migrate_from_registry.py` that reads Projects, Subscriptions, and FormSubmissions from the Registry PostgreSQL database and writes them to the corresponding DynamoDB tables.
2. WHEN migrating a Registry Project, THE Project_Registry SHALL map `id` to a ULID-formatted string, preserve all other fields, and set `migrated_from = "registry"` and `migrated_at` to the migration timestamp.
3. WHEN migrating a Registry Subscription, THE Project_Registry SHALL preserve `person_id`, `project_id`, `status`, `subscription_date`, `is_active`, and `notes`.
4. WHEN migrating a Registry FormSubmission, THE Project_Registry SHALL preserve `project_id`, `person_id`, and `responses`.
5. WHEN a migration record already exists in DynamoDB (idempotency check by original Registry ID), THE migration script SHALL skip that record and log a warning — it SHALL NOT overwrite existing data.
6. THE migration script SHALL log a summary on completion: total records processed, records written, records skipped, and records failed.
7. IF a migration record fails to write to DynamoDB, THE migration script SHALL log the error and continue processing remaining records — it SHALL NOT abort the entire migration.

### Requirement 15: Observability and Security Monitoring

**User Story:** As a platform engineer, I want the service to emit structured logs and CloudWatch alarms, so that operational issues and security events are detected promptly.

#### Acceptance Criteria

1. THE Project_Registry SHALL emit all logs as structured JSON via `structlog`, including `event`, `timestamp`, `level`, `service`, `correlation_id`, and `duration_ms` on application service boundaries.
2. THE Project_Registry SHALL define CloudWatch metric filters and alarms in its CDK stack for: error rate (≥10 in 5 min), critical errors (≥1 in 1 min), auth failures (≥100 in 1 hour), access denied spikes (≥30 in 5 min), and slow operations p95 (>1000ms in 5 min).
3. THE Project_Registry SHALL route all CloudWatch alarms to SNS → Slack channel `C0AE6QV0URH` via the `SLACK_BOT_TOKEN` secret.
4. THE Project_Registry SHALL encrypt its CloudWatch log group with the platform KMS key (`alias/ugsys-platform`) — satisfying `CKV_AWS_158`.
5. WHEN a request is rate-limited, THE Project_Registry SHALL return HTTP 429 with `Retry-After` header and response headers `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset`.
6. THE Project_Registry SHALL remove the `Server` response header from all responses to prevent technology fingerprinting.
7. THE Project_Registry SHALL set security response headers on every response: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload`, `Content-Security-Policy: default-src 'none'; frame-ancestors 'none'`, `Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy: camera=(), microphone=(), geolocation=(), payment=()`, `Cache-Control: no-store, no-cache, must-revalidate` on `/api/*` routes.

### Requirement 16: CI/CD Pipeline

**User Story:** As a platform engineer, I want the service to have a complete CI/CD pipeline matching the platform standard, so that every change is automatically validated and safely deployed.

#### Acceptance Criteria

1. THE Project_Registry SHALL have a `ci.yml` workflow that runs on every push to `feature/**` and every PR to `main`, executing in order: `lint` (ruff), `typecheck` (mypy strict), `test` (pytest, 80% coverage gate), `sast` (Bandit with `bandit -r src/ -c pyproject.toml -ll`), `sast-semgrep` (p/python, p/security-audit, p/owasp-top-ten, p/secrets), `dependency-scan` (Safety, advisory), `sbom` (CycloneDX → Trivy CRITICAL/HIGH, blocks merge), `secret-scan` (Gitleaks, `fetch-depth: 0`), `arch-guard` (grep for layer boundary violations), and `notify-failure` (Slack on any failure).
2. THE Project_Registry SHALL have a `deploy.yml` workflow that runs only on merge to `main`, requires `environment: prod` gate approval, uses OIDC auth via `AWS_ROLE_ARN` (no static AWS keys), and sends a Slack success/failure notification.

### Requirement 20: Circuit Breaker — External Service Resilience

**User Story:** As a platform engineer, I want the service to fast-fail when Identity Manager is unavailable, so that Lambda execution time is not wasted on 5-second timeouts and users get a quick error response instead of hanging.

#### Acceptance Criteria

1. THE Project_Registry SHALL wrap all `IdentityManagerClient` HTTP calls with a Circuit Breaker that tracks consecutive failures and transitions through three states: `CLOSED` (normal), `OPEN` (fast-fail), and `HALF_OPEN` (probe).
2. WHEN the Circuit Breaker records 5 consecutive failures from `IdentityManagerClient`, THE Project_Registry SHALL transition the circuit to `OPEN` state and log the transition at `warning` level with the service name, failure count, and cooldown duration.
3. WHILE the Circuit Breaker is in `OPEN` state, THE Project_Registry SHALL immediately raise `ExternalServiceError` with `user_message = "Service temporarily unavailable, please try again later"` and error code `SERVICE_UNAVAILABLE` — without making any HTTP call to Identity Manager.
4. AFTER 30 seconds in `OPEN` state, THE Circuit Breaker SHALL transition to `HALF_OPEN` state and allow one probe request through to Identity Manager.
5. IF the probe request in `HALF_OPEN` state succeeds, THE Circuit Breaker SHALL transition back to `CLOSED` state and log the recovery at `info` level.
6. IF the probe request in `HALF_OPEN` state fails, THE Circuit Breaker SHALL transition back to `OPEN` state for another cooldown period.
7. THE Circuit Breaker SHALL be defined as an ABC in `src/domain/repositories/circuit_breaker.py` with an in-memory implementation in `src/infrastructure/adapters/in_memory_circuit_breaker.py` — following the same port/adapter pattern as all other outbound interfaces.
8. THE Circuit Breaker SHALL log every state transition (`closed → open`, `open → half_open`, `half_open → closed`, `half_open → open`) with `structlog` including the service name.
9. THE Circuit Breaker SHALL be wired in `src/main.py` lifespan with `failure_threshold=5` and `cooldown_seconds=30`, and injected into `IdentityManagerClient` via constructor.

### Requirement 31: Distributed Tracing with AWS X-Ray

**User Story:** As a platform engineer, I want distributed tracing across all outbound calls and application service methods, so that I can diagnose latency bottlenecks and trace requests end-to-end across services.

#### Acceptance Criteria

1. THE Project_Registry SHALL instrument all outbound AWS SDK calls (DynamoDB, EventBridge, S3) with AWS X-Ray tracing using `aws-xray-sdk-python` patching.

3. THE Project_Registry SHALL create custom X-Ray subsegments for each application service method (e.g., `ProjectService.create`, `SubscriptionService.subscribe`) to provide method-level latency visibility.
4. THE Project_Registry SHALL annotate X-Ray segments with `service.name = "ugsys-projects-registry"`, `service.version`, and `environment`.

7. WHEN X-Ray SDK is unavailable or tracing is disabled (e.g., in tests or local dev), THE Project_Registry SHALL fall back gracefully without raising exceptions — tracing must be non-blocking.
8. THE Project_Registry SHALL configure X-Ray sampling rules to trace 100% of error responses and 5% of successful responses in production (configurable via environment variable `XRAY_SAMPLING_RATE`).
9. THE Project_Registry SHALL add `aws-xray-sdk` to `pyproject.toml` dependencies.
